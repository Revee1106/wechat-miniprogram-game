# Admin Console Compact Workbench Design

**Date:** 2026-04-02

**Goal**

将控制台的事件配置、境界配置、洞府配置统一改造成紧凑式双栏工作台，减少长滚动页面，统一采用“小列表选择 + 右侧局部编辑 + 下拉新增”的交互方式。

## Problem

当前控制台虽然已经支持多类配置，但仍然以大卡片、独立编辑页和长表单为主。用户在频繁切换对象或修改多个子模块时，需要反复滚动和跳转，尤其是事件配置和洞府等级配置，单屏信息密度偏低，操作成本高。

## Design Summary

控制台改用统一的 `ConfigWorkbench` 交互模型：

- 左侧为紧凑清单区，负责搜索、筛选、分组、选择对象、下拉新增。
- 右侧为详情编辑区，只展示当前选中对象。
- 右侧内部使用一级标签或二级标签切分子配置，避免整页铺开全部字段。
- 顶部和底部只保留必要操作：保存当前项、校验配置、重载运行时。

## Shared Interaction Model

### Registry Pane

- 以小条目列表替代大卡片。
- 每个条目只显示高价值摘要，例如名称、类型、风险、等级、启用状态。
- 条目点击后直接切换右侧详情，不再跳转独立编辑页。
- 新增对象通过下拉框选择模板或归类后创建。

### Detail Pane

- 右侧始终只编辑一个对象。
- 使用标签页组织内容，避免长表单。
- 切换标签时保留本地未保存状态。
- 使用粘性操作条固定保存、校验、重载，减少来回滚动。

### Input Constraints

- 所有可中文展示的字段统一显示中文标签，不直接暴露内部字段名。
- 所有可枚举或可推导的字段优先使用下拉、单选、多选标签、步进器，不使用自由文本输入。
- 仅保留必要文本输入：
  - 名称类
  - 描述类
  - 事件正文与选项文案
  - 无法预置枚举的自由文本

## Per-Page Design

### Event Configuration

- 左侧列表显示事件名、事件类型、风险等级、触发方式、启用摘要。
- 顶部保留搜索、事件类型筛选、风险等级筛选。
- 新增事件通过下拉选择事件类型后创建，自动套入默认模板。
- 右侧标签拆分为：
  - 基础信息
  - 触发规则
  - 前置条件
  - 选项与结果
- `选项与结果` 内部再使用紧凑列表切换各个选项，避免把所有选项全部展开。

### Realm Configuration

- 左侧列表按大境界分组展示，条目显示名称、层级、启用状态。
- 新增境界通过下拉选择所属大境界。
- 右侧标签拆分为：
  - 基础信息
  - 突破配置
- 排序操作尽量放在左侧列表内完成。

### Dwelling Configuration

- 左侧列表展示设施名、设施类别、最高等级、等级数。
- 不开放设施 ID 新增，仅编辑现有设施。
- 右侧标签拆分为：
  - 设施信息
  - 等级配置
  - 特殊效果
- `等级配置` 内部使用 `Lv1 / Lv2 / Lv3 ...` 小标签切换当前等级。
- 新增等级通过下拉选择下一等级并创建，不再纵向铺开全部等级表单。

## Reusable Frontend Pieces

计划抽取以下通用组件：

- `ConfigWorkbench`
- `RegistryPane`
- `RegistryList`
- `RegistryToolbar`
- `DetailTabs`
- `StickyActionBar`
- `InlineStat`

这些组件只负责交互壳和布局，不改动现有后端接口协议。

## Compatibility

- 保持现有后台 API 不变。
- 保持现有事件、境界、洞府保存逻辑不变。
- 重构范围集中在 `admin-console/src`。
- 现有中文文案和枚举映射优先复用，必要时补充字段映射层。

## Risks

- 事件编辑页面字段最多，若拆分不彻底，仍可能出现局部滚动过长。
- 事件选项和洞府等级的本地未保存状态需要稳定保留，避免切换标签后丢失输入。
- 现有测试多基于旧页面文本，需要同步调整测试断言。

## Validation Plan

- 更新现有 Vitest 页面测试，覆盖三块配置页的新入口与主路径。
- 至少验证：
  - 导航可进入三块配置页
  - 左侧列表可切换当前项
  - 下拉新增可创建草稿项
  - 右侧标签切换后内容正常显示
  - 保存、校验、重载按钮仍可触发既有逻辑
- 最终执行 `npm test` 和 `npm run build`，确保静态产物可直接被后台服务。
