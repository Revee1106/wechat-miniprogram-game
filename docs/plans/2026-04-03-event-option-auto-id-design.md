# 事件选项自动编号设计

## 背景

当前管理台在“事件配置 -> 选项与结果”里新增多选项时，会创建一个 `option_id` 为空的选项。用户必须手工填写编号，否则保存时会被“每个结果配置都需要编号和文案”拦截。这个流程和事件配置里的其他自动化规则不一致，也容易产生漏填和重复编号。

这次改动的目标是把多选项模式的选项编号改成前端自动生成，并在界面中只读展示，避免人工维护编号。

## 目标

- 在“事件配置 -> 选项与结果”中新增选项时，立即自动生成 `option_id`
- 自动编号在界面中可见，但不可编辑
- 两个事件编辑入口保持一致：
  - `EventEditorPage`
  - `EventListPage`
- 单一结果模式保持现有 `${event_id}_default` 规则不变

## 非目标

- 不重写已有旧选项的编号
- 不在后端新增编号生成逻辑
- 不在保存时重新批量改写选项编号

## 方案对比

### 方案 A：新增时前端立即生成编号

- 编号格式：`${event_id}_option_${n}`
- 若当前事件编号为空，使用保底前缀：`event_option_${n}`
- 生成时跳过当前已占用的编号
- 选项编号输入框改为禁用展示

优点：

- 新增后即可看到最终编号
- 保存前数据已经完整
- 改动集中在现有前端编辑流，不影响后端接口

缺点：

- 需要两个事件编辑入口同时接入相同规则

### 方案 B：保存时再补编号

优点：

- 改动面小

缺点：

- 与“新增了选项则自动生成编号”的需求不符
- 编辑阶段看不到最终编号

### 方案 C：后端生成编号

优点：

- 服务端统一

缺点：

- 前端保存前拿不到编号
- 需要调整现有前端保存和更新逻辑

## 采用方案

采用方案 A。

## 设计细节

### 编号规则

- 仅用于多选项模式下的新选项
- 格式固定为 `${event_id}_option_${n}`
- `n` 从 `1` 开始递增
- 若当前列表中已存在某个编号，则继续找下一个未占用序号
- 若 `event_id` 为空，则使用 `event` 作为前缀，得到 `event_option_${n}`

### 编辑规则

- `option_id` 在界面中只读展示，不允许手工修改
- 现有老选项沿用原编号
- 新增选项从创建开始就带编号

### 一致性要求

- `EventEditorPage` 和 `EventListPage` 的 `handleAddOption()` 必须复用同样的编号生成规则
- `EventOptionEditor` 只负责展示，不再允许通过输入框修改 `option_id`

## 测试策略

### 组件测试

- `EventOptionEditor.test.tsx`
- 断言“选项编号”输入框为禁用状态

### 页面测试

- `EventEditorPage.test.tsx`
- 打开“选项编排”，点击“新增选项”后，新增项自动带上 `${event_id}_option_${n}` 风格编号

- `EventListPage.test.tsx`
- 紧凑工作台入口下新增选项，同样自动生成编号并展示为禁用

## 风险与控制

- 风险：两个页面各自维护 `createEmptyOption()`，容易再次分叉
  - 控制：新增一个局部辅助函数，最小范围内复用同一规则
- 风险：事件编号为空时生成的保底前缀不稳定
  - 控制：统一使用 `event_option_${n}`，不在后续编辑时自动回写
