# 展示本地化映射层设计

## 目标

建立一层统一的展示映射，让系统内部继续使用稳定的英文键、英文错误码和英文持久化字段，但所有面向人的界面最终都显示中文描述。覆盖范围包括：

- 玩家侧小游戏 canvas 页面
- 玩家侧旧版小程序页面
- 运营侧 admin console
- 后续关系型数据库落库后的配置与接口演进

## 现状问题

当前系统的人机界面文案来源分散：

- 运行时实体通常依赖 `display_name`
- API 错误直接透传 `str(error)`
- 前端部分页面通过正则匹配英文错误再手工翻译
- 兜底逻辑会把英文 key 直接显示给玩家或运营

这会带来三个问题：

1. 后端英文错误会直接漏到前台
2. 同一类概念在不同端的中文不一致
3. 后续切到关系型数据库后，如果库中只存英文键，展示层没有统一映射就会持续泄漏内部标识

## 设计原则

1. 内部标识和持久化字段保持英文
2. 面向人的展示文案统一走映射
3. 错误展示优先使用错误码，不依赖英文句子匹配
4. API 响应允许继续携带英文内部键，但必须同时支持展示层本地化
5. 对旧数据和旧接口保留兼容兜底，避免一次性重构全部配置

## 目标架构

### 1. 统一错误码层

后端核心循环错误增加结构化字段：

- `code`
- `params`
- `message`

其中：

- `code` 是稳定英文错误码，例如 `core.time.not_enough_spirit_stones`
- `params` 传递动态参数，例如 `facility_id`
- `message` 保留英文内部描述，便于日志和排查

API 层不再只返回纯字符串 `detail`，而是返回结构化错误详情。前端接收到结构化详情后，统一通过错误码映射生成中文文案。

### 2. 前端展示映射层

前端建立统一展示工具，至少覆盖：

- 错误码到中文提示
- 资源 key 到中文名
- 设施 id 到中文名
- 境界 key 到中文名
- 状态 key 到中文名

规则如下：

- 优先使用接口直接返回的中文展示字段，例如 `display_name`
- 若展示字段缺失，则回退到本地映射
- 若映射仍缺失，最后才展示内部键

这样既兼容当前配置中已存在的中文 `display_name`，也兼容未来数据库只存英文键的模式。

### 3. 控制台展示映射层

admin console 不改变“内部标识可见”这一事实，但所有默认展示、fallback 标题、概要摘要、错误提示都优先使用中文映射，不再把英文内部值直接当作用户可读文本。

适用场景包括：

- `display_name` 缺失时的 fallback label
- major realm 等枚举字段
- 资源、设施、状态等摘要拼装
- 后端结构化错误的中文显示

### 4. 数据库存储策略

后续切换关系型数据库时：

- 表中存英文 `key` / `code`
- 中文展示文案来自配置字段或本地化表
- 不把中文作为唯一业务标识

推荐未来演进为：

- `resource.key = spirit_stone`
- `facility.facility_id = spirit_field`
- `realm.key = qi_refining_early`
- `localized_text(domain, key, locale, text)`

本次实现不直接引入多语言表，但代码结构要能承接这种模式。

## 第一阶段实现范围

第一阶段只做最能立刻消除英文泄漏、同时为后续演进打基础的部分：

1. 后端核心循环错误结构化
2. 玩家前端统一错误码中文映射
3. 玩家前端常见 key 的中文 fallback 映射
4. admin console API 错误中文映射和关键 fallback 映射

不在第一阶段完成的内容：

- 全量配置改为只保留英文 key
- 数据库存储结构改造
- 完整多语言系统

## 关键技术决策

### 错误码优先，不再依赖英文句子正则

旧页面里已经有 `translateDwellingActionError()` 这类英文正则匹配。第一阶段之后，这类逻辑应逐步收敛到统一错误码映射工具中，避免同一条错误在不同页面重复维护。

### 展示映射是兜底，不强制覆盖 display_name

当前配置和 API 已经大量返回中文 `display_name`。这些字段仍然保留并优先使用。映射层主要负责：

- 缺省 fallback
- 内部键直出拦截
- 未来数据库英文存储的承接

### API 返回结构化 detail，前端保留兼容旧字符串

为避免一次性改动所有客户端，前端错误解析按以下顺序处理：

1. 结构化 `detail.code`
2. 旧字符串 `detail`
3. HTTP 状态码默认文案

## 测试策略

需要新增或修改以下测试：

- 后端：错误对象结构化序列化
- 前端小游戏：英文错误码映射成中文 toast
- 旧版小程序页面：结构化错误映射和 fallback key 映射
- admin console：API 错误和 fallback 标签映射

## 风险与控制

### 风险 1：旧页面依赖 `error.message` 字符串判断

控制方式：

- 前端统一把结构化错误重新包装成 `Error`
- `error.message` 始终给中文文案
- 同时附加 `error.code`，供页面后续替换掉正则判断

### 风险 2：一次性替换全部错误字符串成本过高

控制方式：

- 第一阶段只覆盖真正玩家/运营可见的高频错误
- 旧字符串保留兼容映射
- 后续逐步把服务层英文字符串替换为结构化错误码

### 风险 3：前后端重复维护映射

控制方式：

- 第一阶段接受轻量重复，优先止血
- 第二阶段再考虑抽公共配置或生成脚本

## 验收标准

达到以下标准即可认为第一阶段完成：

1. 玩家前台不再直接出现英文错误提示
2. 控制台默认展示不再把英文 key 当作用户文案
3. 新增错误优先通过错误码和映射接入
4. 代码结构已经能承接“数据库只存英文键，界面统一中文展示”的模式
