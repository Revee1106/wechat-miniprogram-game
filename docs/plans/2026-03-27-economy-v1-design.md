# Economy V1 Design

日期：2026-03-27

## 目标

围绕 [经济系统v1.md](/E:/game/docs/plans/经济系统v1.md) 先落地一个最小闭环版本，优先完成：

- 局内资源口径统一的基础骨架
- 单局资源清零规则
- 轮回点结算
- 寿命作为最高级约束资源的接入
- 事件与突破对新经济口径的兼容

本阶段不展开：

- 洞府经营细化
- 炼丹炼器完整玩法
- 战斗掉落与战斗消耗
- 经济控制台

## 设计原则

### 1. 领域模型与存储解耦

当前实现可以继续落在本地配置与内存运行时，但模型设计必须兼容未来迁移到 MySQL。

因此需要明确分层：

- 领域层只关心统一模型
- repository 层负责把本地配置映射成领域模型
- 未来如果迁到 MySQL，只替换 repository 和 DTO 映射，不改业务服务层

### 2. 本地配置结构不强行等于未来表结构

当前本地配置为了可维护，可以更偏文档化和嵌套结构。

未来 MySQL 为了查询、约束和后台维护，应当拆成更范式化的关系表。两者不需要一一同形，但字段语义必须一致。

### 3. 先做闭环，不做全量资源铺设

资源体系会按文档方向设计成可扩展结构，但首版只落最小必要资源，不一次性把所有灵植、灵矿、材料、丹方都接入。

## 资源模型

### 1. 资源定义目录

新增统一资源定义模型：

- `ResourceDefinition`
  - `key`
  - `display_name`
  - `category`
  - `tier`
  - `rarity`
  - `stackable`
  - `tags`

这里的重点是，像灵植不是一个简单整数，而是一类可继续扩展的资源定义。例如：

- `spirit_grass_basic`
- `red_root`
- `nine_leaf_ginseng`

它们都可以属于：

- `category = herb`
- 但各自有不同 `rarity`、`tier` 和 `tags`

### 2. 局内资源持有结构

新增局内资源栈模型：

- `RunResourceStack`
  - `resource_key`
  - `amount`

运行时的资源持有不再依赖无限扩散的散字段，而是逐步向统一资源键收口。

### 3. 本阶段实际启用的最小资源集合

首版先启用以下资源：

- `spirit_stone`
- `basic_herb`
- `basic_ore`
- `basic_breakthrough_material`
- `rare_material`

其中：

- `修为` 继续保留在 `character`
- `寿命` 继续保留在 `character`
- `轮回点` 继续保留在 `player_profile`

这样可以避免一次把整个运行时状态重构掉，但新经济口径已经建立。

## 资源分层映射

本阶段按文档先建立层级概念，不要求一次铺满所有资源：

- 一级资源
  - 灵石
  - 修为
- 二级资源
  - 普通灵植
  - 普通灵矿
  - 普通晋级材料
- 三级资源
  - 稀有材料
- 四级资源
  - 寿命
- 局外资源
  - 轮回点

## 运行规则

### 1. 寿命

寿命继续保留在 `character` 上，不并入普通资源栈。

本阶段寿命的职责：

- 时间推进时持续消耗
- 事件与突破中继续允许变化
- 进入轮回点结算维度

本阶段不新增主动“燃寿”操作入口，只为后续扩展预留规则接口。

### 2. 单局清零

当玩家死亡或转生时：

- 局内资源栈全部清零
- 灵石不继承
- 材料不继承
- 局内产物不继承
- 修为不继承

局外只保留：

- 轮回点
- 非库存型永久进度的预留位置

### 3. 轮回点结算

轮回点先固定写死在后端，不进入控制台。

结算维度：

- 本局最高境界
- 存活时间
- 稀有资源获取数
- 特殊事件完成数

这次只要求形成稳定、可解释、可测试的固定结算公式，不做配置化。

## 与现有系统的衔接

### 1. 事件系统

事件结算会逐步从旧的散字段口径收敛到统一 `resource_key`。

本阶段要求：

- 保留对旧配置的兼容
- 支持事件写入统一资源栈
- 允许部分事件继续通过旧字段工作

### 2. 突破系统

突破继续读取：

- 修为
- 灵石

同时预留未来接入普通/高阶晋级材料的结构位置，但本阶段不强制所有境界使用材料门槛。

### 3. 转生系统

转生结算将成为局内清零和轮回点结算的主入口。

需要保证：

- 本局运行结果先结算出轮回点
- 玩家 profile 获得轮回点
- 新 run 创建时不继承旧局库存

## 存储设计约束

### 当前阶段

先采用本地配置文件：

- `config/economy/resources.json`
- `config/economy/settlement.json`

由 repository 读取并映射成领域模型。

### 未来 MySQL 迁移方向

未来关系型存储建议拆分为：

- `resource_definition`
- `resource_tag`
- `settlement_rule`
- `run_resource_stack`
- `player_profile_currency`

本阶段代码实现必须避免把 JSON 结构写死进服务层逻辑。

## 测试范围

需要覆盖：

- 资源定义目录加载
- 局内资源栈读写
- 单局清零
- 轮回点结算
- 事件奖励写入统一资源口径
- 转生后局外保留、局内清零
- 旧事件配置兼容

## 实施顺序

1. 建立经济领域模型与本地配置 repository
2. 给 `RunState` 和 `PlayerProfile` 接入经济字段
3. 实现资源目录与资源栈兼容层
4. 实现轮回点结算与局内清零
5. 把事件与突破接入新口径
6. 补测试与进度文档
