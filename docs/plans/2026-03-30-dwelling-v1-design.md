# 洞府系统 V1 设计确认

**日期：** 2026-03-30

## 目标

本次实现按 [问道长生_洞府系统_v_1.md](/E:/game/docs/plans/%E9%97%AE%E9%81%93%E9%95%BF%E7%94%9F_%E6%B4%9E%E5%BA%9C%E7%B3%BB%E7%BB%9F_v_1.md) 的首版目标推进，但控制在“洞府主循环先完整落地，炼丹闭环暂缓”。

交付目标是：

- 玩家能在单局内查看洞府设施状态
- 玩家能建造与升级首版 4 个核心设施
- 推进时间时会自动进行洞府月度维护与产出结算
- 洞府页能展示设施、成本、产出、停摆信息和最近一次月结算
- 炼丹房可建造、可维护、可显示“功能已解锁”，但不实现完整炼丹系统
- 产出一份进度文档，明确本次完成与未完成范围

## 本次范围

### 实装

- 灵田
  - 建造 / 升级
  - 月维护扣灵石
  - 产出基础灵植
- 矿洞
  - 建造 / 升级
  - 月维护扣灵石
  - 产出基础矿材
- 聚灵阵
  - 建造 / 升级
  - 月维护扣灵石
  - 月度固定修为收益
  - 继续参与突破辅助加成
- 炼丹房
  - 建造 / 升级
  - 月维护扣灵石
  - 作为功能设施占位，显示已解锁状态

### 不在本次范围

- 完整炼丹系统
- 丹方配置与丹药产物
- 炼器系统
- 洞府布局 / 地块 / 相邻加成
- 建筑耐久与修复流程
- NPC / 弟子 / 仆从系统
- 自动售卖与复杂生产链

## 数据与行为设计

### 运行态

`RunState` 新增洞府设施列表和最近一次月结算结果。

设施状态分为：

- `unbuilt`
- `active`
- `stalled`
- `max_level`

每个设施保存：

- 设施标识与名称
- 设施类型
- 当前等级
- 最大等级
- 当前状态
- 建造成本 / 下一次升级成本
- 当前维护费
- 当前月产出预览
- 功能解锁说明

### 月度结算

时间推进时执行顺序调整为：

1. 月份 +1
2. 寿元 -1
3. 洞府月度维护与产出结算
4. 事件冷却递减
5. 事件抽取

月结算规则：

- 已建造设施逐个检查维护费
- 维护费能支付：扣费并发放产出 / 功能效果
- 维护费不能支付：本月停摆，不发放产出
- 结算结果写回 `run.result_summary` 之外的专用洞府结算字段，供洞府页展示

### 突破联动

突破辅助不再简单依赖裸 `dwelling_level`，改为由 `聚灵阵` 当前等级提供额外加成。

`dwelling_level` 保留，用作洞府总览等级展示，定义为“当前已建造设施中的最高等级”。

## 接口设计

沿用现有 `run` API 风格，新增：

- `POST /api/run/dwelling/build`
- `POST /api/run/dwelling/upgrade`

输入为：

- `run_id`
- `facility_id`

返回值统一直接返回最新 `RunState`。

## 前端设计

洞府页改为真正的操作页，包含：

- 洞府总览卡
- 最近一次月度结算摘要
- 设施列表
- 每个设施的等级、状态、维护、产出、功能说明
- 建造 / 升级按钮
- 炼丹房占位提示

视觉风格保持现有卷轴系页面，不单独引入新视觉体系。

## 测试设计

后端至少覆盖：

- 初始运行态携带 4 个首版设施
- 建造与升级会正确扣费并更新状态
- 推进时间时会触发洞府月结算
- 维护不足时设施停摆
- 聚灵阵会提供修为收益与突破加成
- 新 API 返回结构正确

前端至少覆盖：

- 洞府页包含设施列表与操作按钮
- `run-store` 新增洞府建造 / 升级动作
- 页面读取并展示月结算结果
