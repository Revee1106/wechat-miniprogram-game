# 控制台结果资源编辑器设计

## 目标

把事件结果里的“资源变化”从自由文本输入改成面向策划/运营的结构化编辑器。操作员只需要选择中文资源名称并填写增减数值，不再接触 `spirit_stone:+1` 这类技术表达。

## 现状问题

- `ResultPayloadEditor` 目前用多行文本框编辑 `resources`。
- 文本格式依赖人工记忆 key 和冒号语法，容易写错。
- 资源 key 还存在历史别名，例如 `herbs` / `herb`、`iron_essence` / `ore`，进一步增加维护成本。

## 方案

### 1. 结果资源改为结构化条目编辑

- 每条资源变化由 `资源下拉 + 数值输入 + 删除按钮` 组成。
- 底部提供 `新增资源变化`。
- 下拉只允许选择预设资源，不支持自由输入。
- 同一个资源不能重复添加。
- 数值支持正负；值为 `0` 的条目不进入最终 payload。

### 2. 中文资源目录作为唯一展示来源

- 新增资源目录工具，统一维护：
  - 资源 key
  - 中文名称
  - 历史别名归一化
- 控制台只展示中文名称，例如 `灵石`、`药草`、`玄铁精华`。
- 编辑器保存时仍输出后端当前使用的结构化 payload，不修改 API 协议。

### 3. 对历史配置做兼容归一化

- 结果 payload 里如果已有 `herbs`、`iron_essence` 这类旧 key，编辑器加载时统一转换成控制台内部使用的规范 key。
- 保存后统一写回规范 key，减少同义字段继续扩散。

## 首版范围

- 只改结果 payload 里的 `resources` 编辑方式。
- 接入入口：
  - 单一结果编辑器
  - 选项成功结果
  - 选项失败结果
- 不改“前置条件 / 所需资源”的文本输入，这部分后续单独优化。
- 首版仍使用普通下拉；后续资源类型变多时可升级为可搜索下拉。

## 验证

- 为结果资源编辑器增加组件测试，覆盖：
  - 旧资源别名归一化
  - 新增/修改/删除资源条目
  - 输出 payload 正确
- 运行控制台全量测试和构建。
- 运行后端测试，确保规范化后的资源 key 不影响 payload 解析。
