# 境界目标突破配置设计

## 目标

把境界突破配置从“配在当前境界上”改成“配在目标境界上”。这样 `炼气初期` 作为角色初始境界，不再需要编辑突破配置；从 `炼气中期` 开始，每一条境界都描述“突破到该境界”所需的成功率、资源消耗与失败惩罚。

## 方案对比

### 方案 1：仅在控制台隐藏炼气初期突破配置

- 优点：改动最小。
- 缺点：后端仍按“当前境界配置”结算，控制台语义和运行时语义不一致，失败惩罚也没有明确归属。

### 方案 2：整体切换为目标境界配置

- 优点：配置语义清晰，和“目标境界”展示一致，失败惩罚自然挂在目标境界上。
- 缺点：需要调整运行时读取逻辑、控制台表单和现有配置文件。

### 方案 3：新增独立突破规则表

- 优点：扩展性最好。
- 缺点：范围过大，这次没有必要。

## 采用方案

采用方案 2。

## 数据模型

### 后端 RealmConfig

- 保留：
  - `base_success_rate`
  - `required_cultivation_exp`
  - `required_spirit_stone`
  - `lifespan_bonus`
- 新增：
  - `failure_penalty`

`failure_penalty` 先采用与事件结果相近的结构，但这次只允许：

```json
{
  "character": {
    "cultivation_exp": -50
  }
}
```

暂不开放其他惩罚字段。

### 配置文件语义

- `qi_refining_early`：仅表示当前初始境界本身，不承载突破配置。
- `qi_refining_mid`：表示“从炼气初期突破到炼气中期”的配置。
- `foundation_early`：表示“从炼气大圆满突破到筑基初期”的配置。

## 运行时行为

### BreakthroughRequirements

运行时展示的突破需求改为读取“下一境界”：

- `target_realm_key` 仍然是下一境界
- `required_cultivation_exp` 改为下一境界配置
- `required_spirit_stone` 改为下一境界配置

### ProgressionService

突破流程改为：

1. 找到当前境界和下一境界
2. 用下一境界配置读取成功率、修为需求、灵石需求、失败惩罚
3. 校验资源是否足够
4. 计算最终成功率
5. 按现有服务风格判定成功/失败
6. 成功：
   - 扣除灵石
   - 进入下一境界
   - 增加下一境界寿元加成
7. 失败：
   - 不晋级
   - 应用 `failure_penalty`
   - 当前仅支持扣减修为，最低不低于 0

## 控制台设计

### 目标

让突破配置的编辑语义和运行时一致。

### 页面行为

- 对于排序最前的第一条境界：
  - “突破配置”抽屉只显示提示文案
  - 文案为“当前为起始境界，无需配置突破项”
- 对于其他境界：
  - 显示“突破成功率”
  - 显示“突破所需修为”
  - 显示“突破所需灵石”
  - 显示“突破失败惩罚”

### 失败惩罚交互

这次不直接复用完整事件结果编辑器，而是做一个更窄的编辑器：

- 下拉框：
  - `无惩罚`
  - `扣减修为`
- 选择 `扣减修为` 后展示数值输入框
- 控制台展示中文，不暴露底层字段
- 保存时写回：

```json
{
  "character": {
    "cultivation_exp": -N
  }
}
```

## 配置迁移

现有 `realms.json` 使用的是“当前境界配置”语义，需要迁移到“目标境界配置”：

- 把原 `qi_refining_early` 的突破配置移动到 `qi_refining_mid`
- 把原 `qi_refining_mid` 的突破配置移动到 `qi_refining_late`
- 依次类推
- 第一条境界的突破字段重置为默认无效值

这样默认仓库配置与新语义保持一致。

## 测试范围

### 后端

- 运行时突破需求读取下一境界配置
- 突破成功后仍正确进入下一境界
- 突破失败时按目标境界配置扣减修为
- 第一条境界允许没有有效突破配置
- 配置校验拒绝非法失败惩罚结构

### 前端

- 第一条境界打开“突破配置”时只显示提示
- 非第一条境界可以编辑失败惩罚
- 保存后把惩罚写成目标结构

## 范围约束

- 本次不做独立突破规则表
- 本次不开放修为以外的失败惩罚
- 本次不把突破改造成战斗模块
