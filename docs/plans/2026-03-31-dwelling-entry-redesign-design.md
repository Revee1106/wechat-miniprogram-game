# 洞府入口重构设计

日期：2026-03-31

## 背景

当前事件页在选择 `dwelling` 页签后，直接把完整洞府建造内容塞进上半屏。这样虽然能操作，但主流程页面会突然变成大面积长列表，和 `player / resources / cultivation` 的上半屏交互不一致，也削弱了行程页作为“主控页”的节奏感。

同时，洞府系统和炼丹系统都已经有独立页面，但缺少从事件页进入的明确入口，也缺少稳定的“返回行程”按钮，导致页面层级不够清晰。

## 目标

1. 事件页 `dwelling` 页签改成轻量入口区，不再直接承载完整建造列表。
2. 上半屏仅保留必要信息：洞府等级、已建设施数、最近月结算摘要。
3. 上半屏提供两个主入口：
   - `洞府营建`，进入独立洞府页。
   - `炼丹房`，进入独立炼丹页。
4. 洞府页和炼丹页都提供显式 `返回行程` 按钮，避免用户只能依赖系统返回。

## 方案选择

### 方案 A：继续在事件页直接操作

优点：
- 少一次跳转。

缺点：
- 上半屏信息密度过高。
- 事件页职责继续膨胀。
- 洞府和炼丹已经有独立页，重复维护两套主操作区没有必要。

### 方案 B：事件页入口化，独立页承载完整功能

优点：
- 主流程页保持克制。
- 洞府和炼丹的完整交互继续集中在各自页面。
- 后续扩展炼丹、设施细节时不会继续挤压事件页。

缺点：
- 多一次页面跳转。

结论：采用方案 B。

## 交互设计

### 事件页 `dwelling` 上半屏

- 标题保持“静修根基”方向，但信息压缩为一块总览卡。
- 总览卡展示：
  - 当前洞府等级
  - 已建设施数
  - 最近月结算的关键一句摘要
- 下方放两张入口卡或两个主按钮：
  - `洞府营建`
  - `炼丹房`
- 每个入口附一行简短说明，不堆长列表。

### 独立洞府页

- 保留现有完整设施列表、建造、升级能力。
- 在页头增加 `返回行程` 按钮，优先 `navigateBack`，不具备返回栈时跳回事件页。

### 独立炼丹页

- 保留现有炼丹功能。
- 在页头增加 `返回行程` 按钮，行为同上。

## 技术实现

- 前端仅改 `wechat-miniprogram-game-front`。
- 事件页新增：
  - 洞府摘要文案生成
  - `openDwellingPage`
  - `openCraftingPage`
- 洞府页和炼丹页新增：
  - `returnToJourney`
- 测试更新为断言：
  - 事件页存在洞府入口按钮和跳转方法
  - 事件页不再断言直接建造/升级按钮出现在上半屏
  - 洞府页、炼丹页存在 `返回行程` 按钮与方法

## 风险与处理

- 风险：用户从非事件页路径进入洞府页/炼丹页时，返回行为不稳定。
  - 处理：先尝试 `navigateBack`，失败时回退到 `navigateTo('/pages/event/event')`。
- 风险：事件页摘要过于简略，丢失最近月结算信息。
  - 处理：保留一条最近结算摘要，不展示完整列表。

## 验收标准

1. 事件页切到 `dwelling` 后，上半屏不再出现完整设施长列表。
2. 事件页切到 `dwelling` 后，可看到洞府总览与两个跳转入口。
3. 点击 `洞府营建` 可进入洞府页。
4. 点击 `炼丹房` 可进入炼丹页。
5. 洞府页和炼丹页都有 `返回行程` 按钮，并能返回事件页。
